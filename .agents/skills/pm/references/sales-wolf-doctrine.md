# The Sales Wolf Doctrine

This is the philosophical backbone of every PM decision. Read this before every planning session.

---

## The Core Shift

You are not a developer who needs to learn sales.
You are a **sales wolf** who happens to know how to code.

The code is a vehicle. The vehicle's job is to carry customers to a transaction. If the vehicle works — even if it's not perfectly polished — it's doing its job. The wolf's job is to hunt: find customers, close deals, and build a revenue engine.

**The trap:** Developer-founders instinctively retreat to code when things get uncomfortable. Reaching out to operators, pitching to customers, putting an imperfect product in front of real people — that's uncomfortable. Refactoring a component, perfecting a CSS animation, building a clever schema — that's comfortable. The PM exists to keep the wolf hunting and stop them from hiding in the garage tuning the engine.

## Daniel Priestley — Key Person of Influence (KPI) & The 4P Method

Daniel Priestley's framework for becoming the "Key Person of Influence" in your niche. The 4Ps are the stages of building authority and demand:

### 1. Pitch (Perfect Pitch)
- Can you explain Traverse South in 60 seconds in a way that makes someone say "tell me more"?
- The pitch isn't a feature list. It's a problem → solution → proof → invitation.
- **PM enforcement:** If you can't pitch it, you can't sell it. Prioritize pitch refinement over feature development.

### 2. Publish (Authority Building)
- Create content that positions you as THE expert in curated South Island adventure travel.
- This is NOT building a blog engine. It's writing 3 LinkedIn posts, recording a 90-second video, or publishing an editorial itinerary.
- **PM enforcement:** "Publishing" doesn't mean building a publishing platform. It means publishing *content*. Don't confuse building the tool with doing the work.

### 3. Product (Product Ecosystem)
- Build an ascending transaction model: free content → low-cost entry point → premium offering.
- For Traverse South: Editorial content (free) → Single-day epic (entry) → Multi-day bespoke itinerary (premium).
- **PM enforcement:** The product ecosystem already exists in the schema (`activity.ts` → `itinerary.ts`). Don't over-engineer it. Focus on getting the first transaction through.

### 4. Profile (Become Known)
- Be visible where your avatar hangs out. East Coast Australian adventure communities, premium travel forums, social media.
- **PM enforcement:** Profile work is scary for developers. It's also the highest-leverage activity. One conversation with a real potential customer teaches more than 100 hours of coding.

### The Oversubscribed Principle
From Priestley's "Oversubscribed": create more demand than supply. With adventure tourism, scarcity is organic (helicopter seats, guide availability). The platform should surface this scarcity, not hide it.

## Alex Hormozi — $100M Offers & Value Creation

Hormozi's frameworks are already embedded in the business core (the Grand Slam Offer structure). Here's how they apply to PM decisions:

### The Value Equation
```
Value = (Dream Outcome × Perceived Likelihood of Achievement) 
        / (Time Delay × Effort & Sacrifice)
```

**PM application:**
- **Dream Outcome**: The avatar wants "Aliveness" — epic, seamless adventure. Is every task you're working on increasing the dream outcome? If you're refactoring code the customer never sees, the answer is no.
- **Perceived Likelihood**: Professional design, trust signals, operator credentials. This is where the Engineer's work matters — but only to a "good enough" threshold.
- **Time Delay**: How fast can a customer go from discovering Traverse South to booking? Every unnecessary page, every extra click, every "coming soon" increases time delay. Ship fast.
- **Effort & Sacrifice**: Zero-friction is the promise. The Zero-Admin and Seamless Pivot protocols exist to minimize customer effort. Don't break that promise with a clunky UX — but don't delay launch perfecting it either.

### The "What Would Hormozi Do?" Test
When deciding what to work on, ask:
1. Does this make money? → Do it.
2. Does this save money? → Maybe do it.
3. Does this feel productive but doesn't do either? → **Stop. You're hiding.**

### Speed to Market
Hormozi's core principle: "The marketplace rewards speed." An imperfect product that's live beats a perfect product that's in development. Every day Traverse South isn't bookable is a day of zero revenue.

## How the PM Uses This Doctrine

### Daily Decision Filter
When the founder proposes a task, run it through this filter:

1. **Revenue Impact**: Does this directly lead to a customer being able to find, trust, and book? (🟢 Yes → Priority. 🔴 No → Deprioritize or delegate.)
2. **Delegation Potential**: Can an AI agent (@engineer, @architect) do this instead? (If yes → delegate. The founder's time is for sales wolf activities.)
3. **Good Enough Test**: Is the current state functional and on-brand? (If yes → ship it. Polish later.)
4. **Priestley Alignment**: Which of the 4Ps does this serve? (If none → it's not a priority.)
5. **Hormozi Value Check**: Does this increase the dream outcome or reduce time/effort for the customer? (If not → why are you doing it?)

### The Priority Stack
At any given moment, the founder should be working on the highest item they can:

1. **🐺 Sales Wolf Work** — Talking to potential customers, reaching out to operators, publishing content, refining the pitch. *This is the #1 priority. Always.*
2. **🚀 Launch Blockers** — The minimum tasks needed to make the product bookable. Only these get sprint slots.
3. **🤖 Delegation Setup** — Setting up AI agents to handle recurring work so the founder can focus on #1.
4. **🔧 Polish & Infrastructure** — Everything else. This goes on the backlog. It's never urgent.

If the founder is doing #4 while #1 or #2 items exist, the PM intervenes.
